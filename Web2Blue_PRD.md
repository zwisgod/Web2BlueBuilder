# 🎯 项目代号：Web2BlueBuilder - HTML一键转蓝牙APK生成器

你现在是一位拥有 10 年经验的资深 Android 底层架构师和前端/Node.js 全栈工程师。
我的目标是：开发一个开源工具，能将纯 HTML/JS/CSS 前端项目，一键打包成具备 Android 原生蓝牙通信能力的 APK。
为了实现“高效、极低排错成本、高稳定性”，请你严格遵循本 PRD 进行开发。

## 1. 核心技术生死线：蓝牙双协议兼容
前端开发者不懂蓝牙底层，本工具的 Android 壳子必须做好深度桥接封装。
- 必须同时支持两种底层协议的连接与收发：
  - **经典蓝牙 (SPP / RFCOMM)**：针对 HC-05, JDY-31 等老式串口透传模块。
  - **低功耗蓝牙 (BLE / GATT)**：针对主流低功耗芯片。
- 暴露给注入 HTML 的 JS API 必须极简，例如：`BlueApp.connect(mac, type='SPP')`, `BlueApp.send(data)`。

## 2. 核心排错生命线：硬件级实时调试控制台 (DevTools)
这是本项目的最高优先级功能！为了防止成品打包后无法排查单片机通信问题，生成的 APK 必须自带底层调试外挂：
- **触发机制**：在注入的网页端提供一个隐藏触发器（如：连续点击屏幕顶部边缘 5 次），或在 JS API 中提供 `BlueApp.showDebugConsole()`。
- **悬浮日志窗**：弹出一个不依赖外部网络的 Android 原生（或注入的 HTML）半透明悬浮层。
- **实时数据流拦截**：控制台必须能以时间戳形式，实时滚动打印：
  1. 系统级日志：权限申请结果、蓝牙适配器状态、连接断开错误码。
  2. TX/RX 报文：极其清晰地展示向单片机【发送】和从单片机【接收】的原始数据（支持 String 和 HEX 十六进制切换查看）。
- 开发者只需看着这个窗口，就能瞬间判断是 APP 没发出去，还是单片机回的格式不对。

## 3. 技术选型建议要求
考虑到跨平台稳定性和打包自动化（CLI/GUI），请在你的方案中评估：
- 是用纯 Android 原生 Webview 注入 JSInterface 更稳？
- 还是利用 Capacitor / Cordova 开发深度定制的蓝牙插件，再写个 Node.js 打包脚本更方便？
